# MVP 範囲 — RUNdio

## MVP 定義

**社内数名が実際に触れ、PC（LP＋スマホ枠）と iOS Safari の両方で体験できる最小構成。**  
機能は絞るが、**音声品質・UI・安定性は「社内デモ」とは見えない水準**を目標とする。

---

## 目的

- 課題（ソロランのマンネリ、選択肢の不足）と**解決の方向性**を、**動くもの**で示す。**プレゼンの主役は UX**。
- **マルチユーザー**と**サーバー側生成**は、必要に応じて**補足**として説明できる状態にする。

---

## 対象ユーザー

- **初期**: ペルソナは本人＋社内の数名（iOS 会社端末）。
- **同時利用想定**: **約6名**。**アカウント共有はしない**前提。
- **プレゼン視聴者**: PC ブラウザ。**聴衆は技術寄り**（話の主軸は **UX**）。

---

## デモの前提（重要）

- **プレゼンで流す音声**は、**あらかじめアップロードしたファイル**を使う想定でよい（ライブ生成に依存しないデモ設計）。
- **当面はその音声ファイルを 1 本**用意できればよい（ユーザー別の差し替えは必須としない）。
- プロダクト体験としての「30分放送」は **一続きのラジオ番組**イメージ。実装・台本は後述の方針に従う。

---

## 今回のスコープ外（明示・追記）

- **週次「番組表」**まわりの機能・UI（今回は対象外）。
- **オフライン再生・事前ダウンロード**（一旦検討しない）。
- **ログ・個人情報の保持期間**などのガバナンス詳細（社内デモのため一旦検討しない）。

---

## 成功の定義（MVP）

- [ ] サインアップ／サインインができる（Clerk）。
- [ ] 嗜好・目的等のオンボーディングができる。
- [ ] **目標時間または距離**を指定し、**生成ジョブ**を開始できる。
- [ ] **走り始める前**に、**セッション長に見合う**音声が**概ね**用意できるフローが通る。
- [ ] 音声が**自然に近い**品質（チープな機械音声は避ける）。
- [ ] LP に **スマホ枠**があり、操作デモが可能（完全同一フローでも iframe でも可）。
- [ ] 生成・キーは**クライアント外**（サーバー）で説明できる。

---

## スコープ

### MVP に含める

- Clerk 認証、Supabase `users` 同期。
- `runner_profiles` / `broadcast_jobs` の利用（スキーマはマイグレーション済み想定）。
- オンボーディング、セッション作成、待機、再生の**ユーザーフロー**（[user-flows.md](./user-flows.md)）。
- LP（`/`）と埋め込みデモ（`/demo` から本フローへ拡張）。

### MVP に含めない（明示）

- Garmin / Nike Run Club 等の連携。
- 累計距離・連続日数・週次番組表による**コンテンツ差分**（**今回のリリース範囲では週次番組表も含めない**。将来の設計余地は残してよい）。
- 課金・サブスク。
- ネイティブ App Store アプリ（**モバイル Web ファースト**）。

---

## 制約事項

- 外部 API（LLM/TTS）は**社内ネットワーク・セキュリティポリシー**の対象。
- 長尺一括生成は**レイテンシ・コスト**がボトルネック。必要なら**チャンク生成＋結合**や**非同期ジョブ**を検討。

---

## リスクと対策

| リスク | 対策 |
|--------|------|
| デモ中に生成が間に合わない | **アップロード済み音声をデモで使用**する方針を正とする（録音・事前生成のバックアップも可）。 |
| TTS が不自然 | プロバイダ・ボイス・台本テンプレの調整（[ui-ux-direction.md](./ui-ux-direction.md)）。 |
| iOS 自動再生 | 明示タップで再生。 |
