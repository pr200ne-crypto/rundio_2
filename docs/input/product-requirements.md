# プロダクト要件 — RUNdio

## 1. プロダクト概要

| 項目 | 内容 |
|------|------|
| 名称 | **RUNdio**（Running の「ラン」× Radio の「レディオ」） |
| 形態 | **モバイル最適化 Web**（第一環境: **iOS Safari**） |
| デモ | **PCブラウザ**上LP内の**iPhone相当フレーム**で操作 |
| 認証・データ | **複数ユーザー**を想定し、**サーバー側**で生成・保管 |
| 言語 | **日本語のみ**（UI・音声コンテンツ・コピー） |

---

## 2. 体験・コンテンツ方針（合意）

### 2.1 放送の中身

- **一続きのラジオ番組**という想定で設計する。
- **デモ展示**では、**アップロード済みの音声ファイル**を再生する前提でよい（当日のライブ生成失敗に依存しない）。

### 2.2 目標が「距離」のとき

- 実ペースが読めない前提のため、**尺と距離の対応は多少のズレを許容**する。
- **ユーザーに想定ペースは入力させない**。**実装側の定数**で決め打ちする（変更は定数の更新で足りる）。
- 初版の例として **6:00/km** を想定ペースとしてよい（距離からおおよその所要時間・放送尺を算定）。

### 2.3 脚本・パーソナリティ

- **理想**: 同じユーザーが似た条件でも**毎回まったく別の脚本**。
- **実装優先**: **安定したパーソナリティ**（同一キャラ・トーン）で実装しやすい方でもよい。

### 2.4 表現・安全（コンテンツ）

- **下ネタは禁止**。
- **ブランド名**の扱いは**当面 NG なし**（必要に応じて後から制限を足せるようにする）。
- **心拍・Garmin 等なし**前提。励ましは **「いいペースですね」「無理せず」** など**当たり障りのない一般表現**にとどめる。
- **事実っぽい生成の誤り**（地名・店名等）については、**現状の方針のまま**（追加のファクトチェック要件は置かない）。

### 2.5 今回の範囲から外すもの

- **週次番組表**のUI・ロジック（今回はなし）。

### 2.6 プレゼン

- 聴衆は**技術寄り**だが、**主軸は UX**（「走る前に、自分向けのラジオが用意できる」体験の見せ方）。アーキテクチャ説明は**補助**。
- **締めのキャッチや一言**は任意（決まり次第ここに追記）。

## 3. ユーザーストーリー（MVP）

1. **ランナーとして**、自分の好み・走りの目的を登録したい。**なぜなら**放送のトーンと内容を自分向けにしたいから。
2. **ランナーとして**、**目標時間または距離**を入れてセッションを開始したい。**なぜなら**その長さに合った放送を聞きたいから。
3. **ランナーとして**、**走る前に**放送の生成が**だいたい完了**するまで待ちたい。**なぜなら**走行中に生成待ちで体験が途切れるのを避けたいから。
4. **ランナーとして**、**自然に近い音声**で放送を聞きたい。**なぜなら**没入感と継続利用意欲が失われるから。
5. **デモ視聴者として**、LP上で**実際の操作**を追いたい。**なぜなら**アイデアではなくシステムとして理解したいから。

## 5. 非機能要件（プロダクト観点）

| 項目 | 目標 |
|------|------|
| 音声品質 | **機械的・AIっぽい不自然さ**を避ける（高品質TTS、台本設計） |
| 待機体験 | 生成中に**進捗または残り目安**、失敗時**再試行／短縮モード**の説明可能 |
| UI | **スマホ実機で指操作に耐える**タップ領域・余白 |
| ブランド | LPとアプリの**トーン・タイポを揃え**「社内デモ」とは見えない仕上げ |

---

## 6. MVP の合格条件（合意済み）

- 好み・目的・コースのイメージを入力できる。
- **目標時間または距離**でセッション開始できる。
- **セッション長に見合う長さ**の音声が生成される（目安: 30分ラン → 約30分）。**距離目標のみ**のときは**尺の多少のズレを許容**する。
- **走り始める前に生成が概ね完了**している利用フローが成立する。
- 音声は**自然に近い**品質（参考: ElevenLabs「Mei - Friendly, Clear and Soft」系の方向性）。
- **プレゼン本番**は**アップロード済み音声**でデモしてよい（上記フローはプロダクトとしては維持）。
