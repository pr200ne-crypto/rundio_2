# ビジネス要件 — RUNdio

**RUNdio**: ランニングの「ラン」とラジオ（レディオ）を組み合わせたプロダクト名。

## 1. 背景・目的

### 1.1 背景

- ソロランナーにとって、**マンネリ**と**モチベ維持**は継続的な課題である。
- 既存の選択肢（音楽ストリーミング、radiko 等）は優れているが、**走る時間・距離・シチュエーションに最適化された音声**という**別の選択肢**が乏しい。

### 1.2 目的（本プロジェクト）

- **社内の数名が実際に触れられるサービス／システム**を構築し、**課題提起・提案としてプレゼンテーション**できる状態にする。
- 収益化を主目的とはしないが、**エンドユーザー向けプロダクトに近い品質**（音声・UI・安定性）を求める。

### 1.3 ステークホルダー

| 役割 | 説明 |
|------|------|
| プロダクトオーナー／初期ユーザー | 本人（ペルソナの一次ソース） |
| デモ視聴者 | 社内同僚（**技術寄り**想定）。**プレゼンは UX を主軸**にする。 |
| デモ利用者 | 同時**約6名**、**アカウント共有なし** |
| 開発 | 同上またはチーム |

---

## 2. 解決する課題（Problem）

- **誰の**: 主に**一人で走るランナー**（初期は本人）。
- **何が痛いか**: 単調さ、**「もっと走ったらどんな体験が待っているか」**という期待が持ちにくい。
- **どう良くなるか**: 好み・ゴール・セッション長に合わせた**パーソナル「ラジオ」**により、楽しさと継続意欲が高まる。

---

## 3. 価値提案（Value）

- **走行セッションの長さにフィットした音声**（例: 30分ラン → 約30分の放送）。
- **寄り添いトーク**（好みに応じたゴール近くの励まし等）。
- **変化の余地**: 累計・連続日数・週次の「番組」イメージで**リピート時の新鮮さ**（v1で全部実装する必要はない）。

---

## 4. 成功の定義（本フェーズ）

- PCブラウザ上の**LP内スマホ枠**で、**操作に追随するデモ**が可能。
- 社内の**iOS（Safari）**から同一URLにアクセスし、**同等の体験**が再現できる。
- プレゼン時に**音声品質が「プロトタイプのお遊び」に見えない**水準（自然なTTS、破綻した台本の抑制策の説明ができる）。

---

## 5. スコープ外（明示）

- Garmin / Nike Run Club 等との**自動連携**（将来検討）。
- アプリストア配布の**ネイティブ専用アプリ**を前提とした開発（Webファースト）。
- マネタイズ・課金の実装（当面不要）。

---

## 6. 制約・前提

- 外部API（LLM、TTS 等）の利用は**社内方針・ネットワーク**に依存する。運用前に可否を確認する。
- 生成コストは**デモ人数規模**では許容しやすいが、設計上は**サーバー側キー・クォータ**を前提とする。
