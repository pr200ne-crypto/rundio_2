# システム要件（軽量版）— RUNdio

## 1. 想定規模

| 項目 | 想定 |
|------|------|
| 同時利用 | **数名**（社内デモ） |
| データ量 | 音声ファイルが主因。ユーザーあたりセッション数は限定的でもよい |

---

## 2. プラットフォーム

| 層 | 技術（決定） |
|----|----------------|
| フロント | **Next.js**（App Router）、**Tailwind CSS** |
| 認証 | **Clerk**（既存 `demo-implementation` 方針に合わせる） |
| BaaS / DB | **Supabase**（Postgres、将来 Storage で音声） |
| ホスティング | **Vercel** 想定（未デプロイでもよい。文書上の前提） |

第一クライアント: **iOS Safari**。LPデモ: **デスクトップ Chrome / Edge / Safari**。

---

## 3. セキュリティ

- **LLM・TTS の API キーはクライアントに置かない**（サーバー環境変数またはSecrets）。
- ユーザーごとの生成物・設定は **user_id で分離**（Clerk user と Supabase `users` の紐づけ）。
- 個人情報: メール（Clerk）、嗜好テキスト。**社内デモでもプライバシーポリシー案の1枚**はプレゼン用に用意推奨。

---

## 4. 可用性・パフォーマンス（MVP）

- デモ時間帯の**単一障害**に備え、**事前生成済み音声のバックアップ**をローカル／ストレージに持てるとよい。
- **長尺生成**（30分）はタイムアウトしやすい。**非同期ジョブ**＋ポーリングまたはSSE/WebSocketは後続実装で可。

---

## 5. 外部サービス（想定）

| 用途 | 例 | 備考 |
|------|-----|------|
| TTS | ElevenLabs 等 | 品質要件の中心 |
| LLM | OpenAI / Anthropic 等 | 台本生成 |
| オブジェクト保存 | Supabase Storage | 生成音声 |

**社内ネットワーク**からの外向き通信が許可されるか、事前確認。

---

## 6. 監視・ログ（Should）

- 生成ジョブの**失敗理由**をサーバーログに残す。
- クライアントは**ユーザー向けに穏当なエラー文言**。
