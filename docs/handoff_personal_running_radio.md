# ハンドオフ：RUNdio（要件整理メモ）

**プロダクト名 RUNdio**（ラン × レディオ）。表示・ドキュメントはこの表記に統一。

**更新日**: 2026-04-06  
**目的**: オフィス外・別セッションでも会話・実装を続けられるよう、合意した内容を1ファイルに集約する。

---

## 1. プロダクトの核

- **課題**: ソロランのマンネリ。音楽（Amazon Prime）や radiko は楽しいが、**走る時間・シチュに合った音声**という**選択肢**がもっと欲しい。
- **価値**: パーソナライズされた「ランニング特化ラジオ」。**累計距離・連続日数**でエピソード／トーンが変わる、**週ごとに番組表が変わる**イメージあり（MVPに全部入れる必要はない）。
- **体験のキモ**: 好み（例：ビール、銭湯・サウナ）やゴールに合わせた**寄り添いトーク**（例：「あと少しで冷えたビール」）。
- **好奇心**: 「もっと走るとどんな放送が聞けるか」がモチベになる。

---

## 2. 現在の習慣（ペルソナ＝本人）

- Garmin で計測。
- スマホで **Amazon Prime Music** や **radiko** を聞きながら走る。
- **最初のターゲットユーザーは自分自身**（要件の解像度源）。

---

## 3. MVP の合格ライン

- **入力**: 好み・目的のランニング（コースのイメージ等）＋**目標時間または距離**を自分で入れてスタート。
- **出力**: そのセッションに合った長さの放送（例：30分走るなら**約30分の音声**）。
- **品質**: **自然に近い音声**。**ガチな機械音・AIっぽい不自然さはNG**。
- **音声の参考**: ElevenLabs「Mei - Friendly, Clear and Soft」相当（ローカル参照ファイルあり。AIはMP3を直接聞けない場合あり）。
- **生成タイミング**: **走り始める前に、だいたい生成が終わっている**（事前生成）。→ 生成待ちUX（進捗・目安時間）はMVPで設計対象。
- **Garmin / Nike Run Club 等**: **MVPでは必須にしない**。**将来**、走行データ連携で最適化したい。

---

## 4. 聞き方・置き換え

- 本命: **このサービスに完全置き換え**。
- 許容: **Amazon Prime を小さめBGM**にしつつメインは本サービス。

---

## 5. ビジネス文脈（実態）

- **マネタイズは当面の主目的ではない**。
- **社内の数名が触れるサービス／システムを構築し、課題発表・プレゼン用**。
- それでも **一定のプロダクト品質**（音声・UI・安定性・説得力）は必須。「社内向けだから雑」は不可。

---

## 6. 技術・提供形態（合意）

- **MVP機能はシンプル**。**裏側はマルチユーザー＋サーバー側生成を見越す**（アカウント分離、APIキーはサーバー、等）。
- **プレゼン**: 主に **PCブラウザ**。**LP（ランディングページ）内に iPhone 相当の画面枠**を置き、その中で**操作デモ**するイメージ。
- **社内実機**: 会社携帯は **iOS** → **iOS Safari を第一対象**（レイアウト・Safe Area は iPhone 想定）。
- **LP上の「エミュレータ」**: 実機iOSをそのままWebに埋め込むのは一般的でない。**多くは「スマホ枠＋モバイル最適化Web」**でデモ。ネイティブSwiftのみにするとLP内ライブ操作と両立が難しくなりがち → **モバイルWeb一本**が現実解になりやすい（最終スタックは未確定）。

## 7. タスク一覧（続き用・優先順）

※ `[ ]` は未着手／`[x]` は完了として自分で付け替え。

### Phase 0 — 前提の確定（最初にここ）

- [ ] **スタック決定**: LP＋iPhone枠デモを優先するなら **モバイルWeb（Next.js 等）一本**が現実的、とハンドオフにメモあり。採用スタック（フロント／API／DB／ホスティング）を1行で決める。
- [ ] **MVPの線引き**: v1に入れる機能 vs **後回し**（週次番組表・連続日数・累計連動のうち、デモで最低限見せる範囲）を箇条書きで固定。
- [ ] **社内デモの制約**: 社内ネットワーク・VPN・APIキー管理・外部SaaS（ElevenLabs等）が使えるか、確認メモを残す。

### Phase 1 — 要件・設計ドキュメント

- [ ] **ビジネス／プロダクト要件の正式化**: 本ファイルを土台に、`docs/input/` 相当の構成で `business-requirements.md` / `product-requirements.md`（または既存テンプレ）へ転記。
- [ ] **システム要件**: `1_system-requirements-prompt.md` を回す前提なら、非機能（同時接続数名、可用性、ログ）を1枚に書く。
- [ ] **ユーザーストーリー／画面一覧**: オンボーディング → 目標入力 → 生成待ち → 再生 →（履歴）の**画面リスト**と遷移。
- [ ] **アーキテクチャ1枚**: クライアント（LP＋アプリ枠）／BFF or API／認証／ジョブ（生成）／オブジェクトストレージ／外部TTS・LLM。プレゼン用。

### Phase 2 — 生成パイプライン（技術リスクが高いので早めに）

- [ ] **プロトタイプ**: 短い台本 → TTS → 再生まで、**コマンドまたは最小API**で通す（品質・速度・コストの感触）。
- [ ] **30分分の長尺**: 分割生成・結合・タイムアウト、失敗時フォールバックの方針を決める。
- [ ] **APIキー**: サーバー側のみ。レート制限・ユーザー単位クォータの余地を残す。

### Phase 3 — バックエンド・データ

- [ ] **マルチユーザー最小**: ユーザーID、セッション／ジョブ／生成物メタのスキーマ案。
- [ ] **認証**: 社内数名向けの最小実装（メール＋マジックリンク、Authプロバイダ、等）を選定・実装。
- [ ] **ジョブ処理**: 生成リクエストを非同期化（キュー or バックグラウンド）、ステータスAPI（待機画面用）。

### Phase 4 — フロント（品質の見せ所）

- [ ] **LP**: コンセプト・デモ導線（スマホ枠への誘導）。
- [ ] **iPhone枠内UI**: 375/390pt 想定、Safe Area、タップ領域。PCでも操作しやすいこと。
- [ ] **生成待ちUX**: 進捗表示・目安時間・失敗時メッセージ・再試行。
- [ ] **オーディオプレイヤー**: バックグラウンド再生はPWA/Safariの挙動を確認（MVPで必須かどうかも決める）。

### Phase 5 — プレゼン仕上げ

- [ ] **デモシナリオ**: 操作手順を3分以内に収める台本。
- [ ] **社内iOS実機確認**: Safariで同一URLが問題なく動くか。
- [ ] **バックアップ**: 生成失敗時は録音デモ or 事前生成ファイルで見せられるようにする。
